# tho-008 — Demo Data Package Generation And Remote Import Distribution

## Business Objective

Generate and publish demo-data release packages from canonical vendor source media so the WordPress admin importer can download a GitHub-hosted zip whose `media/` payload matches the vendor dataset and preserves vendor-specific media assignments.

## Actors

- **Admin / release engineer** — rebuilds demo media, publishes a GitHub release, updates manifest pointers
- **System** — assembles canonical media tree, builds zips, updates `demos/manifest.json`, and imports the remote pack
- **Tester** — validates that the remote importer succeeds and vendors resolve to correct media after import

## UX Journey / User Actions

1. Maintainer updates demo source assets under `demos/<pack-id>/media-original/` and `vendor-data.json`.
2. Maintainer runs `demos.media.rebuild <pack-id>`.
3. The workflow converts vendor MP4 files to vendor-relative WebM outputs.
4. The workflow assembles canonical `demos/<pack-id>/media/` from non-video source assets plus rebuilt WebMs.
5. Maintainer runs `demos.release <version> --push`.
6. Release script zips `vendor-data.json` plus `media/`, publishes the GitHub release, and updates `demos/manifest.json`.
7. Importer fetches the manifest, downloads the release asset, and imports vendors into WordPress.
8. Maintainer validates admin AJAX import success and correct per-vendor media outcomes.

## Inputs

| Field | Type | Source |
|---|---|---|
| pack_id | string | CLI argument / pack folder under `demos/` |
| version | semver string | release operator |
| vendor-data.json | JSON | canonical demo vendor dataset |
| media-original/ | directory tree | canonical source videos and source non-video assets |
| manifest.json | JSON | published pack registry consumed by importer |

## Outputs

```
DemoPackRelease {
  release_tag: string
  zip_asset_url: string
  manifest_entry: {
    version: string
    release_tag: string
    zip_url: string
  }
  media_tree: directory  // canonical demos/<pack-id>/media/
}
```

## State Transitions

```
media-original/ + vendor-data.json
  → vendor-relative webm tree
  → canonical media/
  → release zip
  → GitHub release asset
  → manifest pointer
  → remote importer download/import
```

## Conditional Logic

- `demos.media.rebuild` requires `vendor-data.json` plus MP4 inputs under `demos/<pack-id>/media-original/` or the latest archival `media-backup-*` tree; absence is fatal.
- WebM conversion must preserve vendor-relative paths. Flattening by basename is invalid.
- `demos.release` packages `media/` if present; falls back to `media-original/` only when canonical `media/` does not yet exist.
- `vendor-data.json` banner/gravatar references must resolve within the packaged `media/` tree.
- New media corrections should be published under a fresh semver/release tag instead of relying on old-tag overwrite behavior.

## Side Effects

- Writes rebuilt WebM files to a converted media tree.
- Replaces or creates canonical `demos/<pack-id>/media/`.
- Publishes or updates a GitHub Release asset.
- Rewrites `demos/manifest.json` version/release URL fields.
- Remote import writes vendors, media attachments, and related CPT/meta state into WordPress.

## Core Contract Definition

```
rebuildDemoMedia(pack_id: string, quality?: string, crf?: string) → CanonicalMediaTree
  // 1. Read demos/<pack_id>/media-original/ and vendor-data.json
  // 2. Convert MP4 files to vendor-relative WebM outputs
  // 3. Assemble canonical demos/<pack_id>/media/
  // 4. Validate vendor-data.json references against media/

publishDemoRelease(version: string) → DemoPackRelease
  // 1. Zip vendor-data.json + media/
  // 2. Publish or update GitHub Release asset
  // 3. Update demos/manifest.json version, release_tag, zip_url

validateRemoteDemoImport(zip_url: string) → ImportValidation
  // 1. Import remote asset into local WP runtime
  // 2. Validate success response and vendor/media outcomes
```

## Compatibility Adapter Behavior

- Current importer is a WordPress plugin flow driven from `demos/manifest.json` and a GitHub-hosted zip asset.
- Release generation is shell-script based and independent of Dokan/Woo business logic.
- Semantic validation still depends on current local WordPress runtime plus importer/admin AJAX flow.

## Default WP Adapter Behavior

- The packaging contract remains the same: `vendor-data.json` + canonical `media/` tree + manifest-discoverable zip URL.
- The importer may evolve away from Dokan-era internals, but release packaging and hosted asset semantics stay stable.
- Validation oracle remains importer success plus correct media assignment outcomes.

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Canonical zip structure | Asset contains `vendor-data.json` and `media/` only |
| Media reference integrity | `vendor-data.json` banner/gravatar references resolve in packaged `media/` |
| Vendor-relative media fidelity | Distinct source vendor videos remain distinct after WebM conversion and packaging |
| Manifest pointer correctness | `demos/manifest.json` points to the fresh release tag and asset URL |
| Remote import success | Import completes with success response and expected vendor counts |
| Semantic media correctness | Imported vendors resolve to correct vendor-specific media instead of shared duplicates |
