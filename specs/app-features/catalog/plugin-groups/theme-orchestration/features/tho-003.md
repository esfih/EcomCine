# tho-003 — Vendor Profile Meta Synchronization and Custom Fields

## Business Objective

Maintain extended vendor profile meta fields (biography, social links, skills/tags, headline) that extend the default Dokan profile data model, synchronizing data between WP user meta and rendering contexts.

## Actors

- **Vendor** — updates their profile (bio, headline, social links, skills) via Dokan dashboard or admin page
- **System** — persists, reads, and syncs extended meta fields; computes completeness
- **Buyer** — sees rendered profile data on store page (social links, headline, skills)

## UX Journey / User Actions

1. Vendor opens Dokan dashboard → Settings → Store Settings
2. Vendor fills in extended fields: headline, skills (tags), social links (LinkedIn, Twitter, Instagram, etc.)
3. On save (`dokan_store_profile_saved`): theme hook or plugin saves extended meta to WP user meta under `tm_*` / `dokan_*` conventions
4. Store page template reads these meta values for headline display, social links section, skills tags
5. Completeness score computed from filled meta fields → displayed in account panel or store profile
6. Admin can view/edit vendor extended meta in WP admin (user edit screen)

## Inputs

| Field | Type | Source |
|---|---|---|
| vendor_id | int | Profile save hook or template context |
| tm_vendor_headline | string | Vendor form input |
| tm_vendor_skills | string[] / comma CSV | Vendor form input |
| tm_social_linkedin | URL string | Vendor form input |
| tm_social_twitter | URL string | Vendor form input |
| tm_social_instagram | URL string | Vendor form input |
| Additional social_* meta | URL strings | Vendor form inputs |

## Outputs

| Output | Description |
|---|---|
| Persisted user meta | WP user meta entries under `tm_*` / `dokan_*` key conventions |
| VendorProfileMeta | Structured object for template/REST consumption |
| Completeness score | Integer 0–100 + list of missing fields |

## State Transitions

```
profile_state:  incomplete → partially_complete → complete
                (determined by completeness computation, not a hard state machine)
meta_value:     empty → populated → updated
```

## Conditional Logic

- Completeness score: weighted count of filled meta fields (e.g., bio=20pts, headline=10pts, social=10pts each, skills=10pts)
- Social links only rendered on store page if non-empty value
- Skills rendered as tag list only if `tm_vendor_skills` is non-empty
- Headline replaces bio fallback in summary views if non-empty
- Admin-only fields (internal notes) stored but not rendered publicly

## Side Effects

- `update_user_meta` per field on profile save
- Completeness score may be cached in user meta (e.g., `tm_completeness_score`) and refreshed on save

## Current Dependencies

| Dependency | Usage |
|---|---|
| Dokan action `dokan_store_profile_saved` | Save hook for extended meta |
| `get_user_meta` / `update_user_meta` | Extended meta read/write |
| `theme/includes/social-metrics.php` | Social link rendering (tho-005) |
| `theme/includes/completeness.php` | Completeness scoring module (tho-005) |
| Dokan profile settings templates | Extended fields injected via Dokan dashboard hooks |

## Core Contract Definition

```
getVendorProfileMeta(vendor_id: int) → VendorProfileMeta

VendorProfileMeta {
  biography: string
  headline: string
  skills: string[]
  social: {
    linkedin: string|null,
    twitter: string|null,
    instagram: string|null,
    ...
  }
  completeness: int    // 0–100
}

saveVendorProfileMeta(vendor_id: int, data: Partial<VendorProfileMeta>) → bool

computeCompletenessScore(vendor_id: int) → CompletenessResult
CompletenessResult {
  score: int
  missing_fields: string[]
}
```

## Compatibility Adapter Behavior

- Extended meta persisted via `dokan_store_profile_saved` action hook
- Data stored in WP user meta with `tm_*` key prefix
- Social and completeness modules in `theme/includes/*.php`
- Dokan profile save hook as persistence trigger

## Default WP Adapter Behavior

- Vendor CPT `tm_vendor`: profile meta stored as CPT post meta (replacing user meta for vendor-specific data)
- Dedicated update endpoint: `POST /wp-json/tm/v1/vendor-profile` (no Dokan dashboard hook needed)
- Completeness score computed from CPT meta fields (same algorithm, different data source)
- `dokan_store_profile_saved` hook replaced by REST endpoint save handler

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Meta persistence | All extended fields saved and read back correctly after profile save |
| Completeness score | Same score value for equivalent meta fill state |
| Social links | Correct URLs rendered per social platform for non-empty values |
| NULL handling | Empty/null fields do not produce display artifacts on store page |
| Admin editability | Extended meta visible and editable in WP admin user screen |
