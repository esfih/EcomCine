# tho-002 — Asset Governance for Vendor Pages

## Business Objective

Control which stylesheets and scripts are loaded on vendor store pages by dequeuing unwanted Dokan/WooCommerce/third-party assets and enqueuing only the cinematic theme's required assets, ensuring clean performance and visual coherence.

## Actors

- **System** — executes during `wp_enqueue_scripts` hook (high priority)

## UX Journey / User Actions

1. Vendor store or listing page loads
2. Theme `wp_enqueue_scripts` callback fires at priority 999 (or equivalent late priority)
3. System checks page context via `dokan_is_store_page()` / `dokan_is_store_listing()`
4. Dequeues known Dokan default styles and scripts (e.g., `dokan-style`, `dokan-scripts`)
5. Dequeues or deregisters unwanted WooCommerce scripts (e.g., `wc-cart-fragments`) where safe
6. Conditionally dequeues Mapbox / Google Maps / Google Fonts if not required by active vendor features
7. Enqueues cinematic theme CSS/JS (main stylesheet, player assets)

## Inputs

| Field | Type | Source |
|---|---|---|
| Page context | bool flags | `dokan_is_store_page()`, `dokan_is_store_listing()`, `tm_is_showcase_page()` |
| Registered asset handles | string[] | WP asset registry |

## Outputs

| Output | Description |
|---|---|
| Dequeued handles | Dokan/WC/third-party handles removed from page |
| Enqueued handles | Cinematic theme CSS/JS handles added to page |

## State Transitions

None — asset registry modification only.

## Conditional Logic

- Dequeue runs **only** on relevant vendor pages (store page, listing, showcase, platform)
- Dequeue `dokan-style`, `dokan-scripts` → replace with cinematic layout assets
- Keep: WC scripts required by tvbm (modal checkout: `wc-checkout`, WC nonces, WC Bookings scripts)
- Keep: `wc-cart-fragments` only if WC cart display is present on page (conditionally)
- Dequeue `mapbox-gl` unless vendor profile has location meta and map feature is active (tho-005)
- Dequeue `google-fonts-*` handles where theme uses self-hosted or system fonts
- Safety check: never dequeue handles that other plugins actively depend on without testing for regressions

## Side Effects

- Page load performance impact (fewer HTTP requests, reduced render-blocking assets)
- Visual regression risk if wrong asset handle dequeued (missing CSS class, broken JS event)
- Console errors if dequeued JS was a dependency of still-loaded JS

## Current Dependencies

| Dependency | Usage |
|---|---|
| `dokan_is_store_page()` | Store page context detection |
| `dokan_is_store_listing()` | Store listing context detection |
| `wp_dequeue_script` / `wp_dequeue_style` | Remove specific handles |
| `wp_deregister_script` / `wp_deregister_style` | Full deregistration where needed |
| Known Dokan asset handles | `dokan-style`, `dokan-scripts`, `dokan-vendor-registration`, etc. |
| Known WC asset handles | `wc-cart-fragments`, `wc-add-to-cart`, etc. |
| WP asset dependency graph | Must not break dependency chains for retained handles |

## Core Contract Definition

```
getAssetPolicy(page_context: PageContext) → AssetPolicy

AssetPolicy {
  dequeue_scripts: string[]    // handles to remove from script queue
  dequeue_styles: string[]     // handles to remove from style queue
  keep_scripts: string[]       // handles explicitly retained (carry-forward for tvbm)
  enqueue: {
    scripts: { handle, src, deps, ver }[]
    styles: { handle, src, deps, ver }[]
  }
}

// Policy is page-context-driven:
// - store_page → full dequeue policy
// - store_listing → listing dequeue policy
// - non-vendor page → no dequeue
```

## Compatibility Adapter Behavior

- Direct `wp_dequeue_*` / `wp_deregister_*` calls inside late `wp_enqueue_scripts` hook (priority ≥ 999)
- `dokan_is_store_page()` + `dokan_is_store_listing()` for context
- Explicit Dokan/WC handle lists maintained in `theme/functions.php` or dedicated include
- tvbm-required WC handles explicitly retained in policy

## Default WP Adapter Behavior

- Asset policy registry: PHP config array `theme/includes/asset-policy.php` mapping `page_context → {dequeue, keep, enqueue}`
- Page context detection via `is_author() || is_post_type_archive('tm_vendor')` (no Dokan functions)
- Same `wp_dequeue_*` / `wp_enqueue_*` calls; Dokan-specific handles no longer present to dequeue (Dokan removed)
- Simplified policy: fewer third-party handles to manage once Dokan/WC dependency chain removed

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Clean store page | Dokan default stylesheet absent from store page source |
| No visual regression | Store page renders correctly with expected cinematic layout |
| WC checkout handles retained | tvbm modal checkout scripts present when modal is active |
| No console dependency errors | No JS errors from missing dependencies on store page |
| Non-store pages unaffected | Asset policy not applied on shop page, product page, blog |
