# tmp-002 — Fallback Media Strategy (Banner Video / Banner Image)

## Business Objective

Resolve an ordered fallback chain for a vendor's banner video and banner image so that the media player always has presentable background media when a full playlist is unavailable.

## Actors

- **System** — called as part of `tm_get_vendor_media_playlist()` construction (tmp-001)

## UX Journey / User Actions

1. After playlist items are resolved (tmp-001), system evaluates the fallback slots
2. Reads `dokan_banner_video` user meta for a playable video URL
3. If empty → `fallbackVideo = ''`
4. Reads vendor banner image via Dokan vendor model `get_banner()` → attachment ID → `wp_get_attachment_url`
5. If Dokan banner model fails → falls back to `dokan_banner` user meta (attachment ID or URL)
6. If nothing resolves → `fallbackImage = ''`
7. Both fallback values returned in `PlaylistPayload`

## Inputs

| Field | Type | Source |
|---|---|---|
| vendor_id | int | Caller (from tmp-001 flow) |

## Outputs

```
FallbackMedia {
  fallbackImage: string    // URL or ''
  fallbackVideo: string    // URL or ''
}
```

## State Transitions

None — pure resolution function. No state written.

## Conditional Logic

- **fallbackVideo chain**:
  1. `get_user_meta($vendor_id, 'dokan_banner_video', true)` → if non-empty → use as URL
  2. Else → `''`
- **fallbackImage chain**:
  1. `dokan()->vendor->get($vendor_id)->get_banner()` → attachment ID → `wp_get_attachment_url(id)` → if non-empty → use
  2. Else → `get_user_meta($vendor_id, 'dokan_banner', true)` → treat as attachment ID → `wp_get_attachment_url` → if resolves → use
  3. Else → `''`

## Side Effects

None — read-only.

## Current Dependencies

| Dependency | Usage |
|---|---|
| Dokan `dokan()->vendor->get($vendor_id)` | Vendor object for `get_banner()` banner attachment resolution |
| `get_user_meta($vendor_id, 'dokan_banner_video')` | Banner video URL |
| `get_user_meta($vendor_id, 'dokan_banner')` | Fallback banner image attachment ID |
| `wp_get_attachment_url` | Resolve attachment ID to URL |

## Core Contract Definition

```
resolveFallbackMedia(vendor_id: int) → FallbackMedia
  // fallbackVideo: user meta 'dokan_banner_video' → URL or ''
  // fallbackImage: Dokan vendor model get_banner() → wp_get_attachment_url
  //               OR user meta 'dokan_banner' → wp_get_attachment_url
  //               OR ''
  // Both values are always strings (never null)
```

## Compatibility Adapter Behavior

- `dokan()->vendor->get($vendor_id)->get_banner()` resolves the banner attachment ID via Dokan vendor model
- `get_user_meta($vendor_id, 'dokan_banner_video', true)` supplies video URL directly
- `wp_get_attachment_url` converts any attachment IDs to URLs

## Default WP Adapter Behavior

- **fallbackImage**: `get_post_thumbnail_id($vendor_cpt_id)` → `wp_get_attachment_url`  
  Eliminates Dokan vendor model entirely; vendor CPT featured image is the canonical banner
- **fallbackVideo**: custom post meta field `tm_banner_video` on vendor CPT (replaces `dokan_banner_video` user meta)
- `wp_get_attachment_url` calls unchanged

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| fallbackImage URL | Same URL returned for equivalent vendor banner state |
| fallbackVideo URL | Same URL returned for equivalent `dokan_banner_video` meta state |
| Empty string | `''` returned when no banner or video is configured — never null or false |
| Fallback chain order | Dokan model used first; user meta used only if Dokan model fails |
