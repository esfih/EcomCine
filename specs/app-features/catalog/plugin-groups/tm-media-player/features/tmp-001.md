# tmp-001 â€” Vendor Playlist Extraction from Biography Shortcodes

## Business Objective

Build a normalized media playlist payload from a vendor's biography shortcodes (`[gallery]` and `[playlist]`) and banner metadata, for consumption by the media player UI.

## Actors

- **System** â€” called by REST endpoint or store page render with a `vendor_id`
- **Vendor** â€” indirect source: biography content (shortcodes) and banner media settings

## UX Journey / User Actions

1. System receives `vendor_id` (from REST request or store page render)
2. Calls `tm_get_vendor_media_playlist($vendor_id)`
3. Attempts `dokan_get_store_info($vendor_id)` for `vendor_biography`; falls back to `get_user_meta($vendor_id, 'vendor_biography', true)`
4. Parses biography string using `get_shortcode_regex(['gallery', 'playlist'])`
5. For each shortcode match: extracts `ids` attribute, resolves each attachment ID via `wp_get_attachment_url`
6. For each resolvable attachment: builds `MediaItem` with type, src, mime, duration, poster, title
7. Skips items where `wp_get_attachment_url` returns false
8. After items: resolves `fallbackVideo` from user meta `dokan_banner_video`
9. Resolves `fallbackImage` from Dokan vendor banner; falls back to `dokan_banner` user meta

## Inputs

| Field | Type | Source |
|---|---|---|
| vendor_id | int | Caller (REST param or template context) |

## Outputs

```
PlaylistPayload {
  items: MediaItem[]
  fallbackImage: string    // URL or ''
  fallbackVideo: string    // URL or ''
}

MediaItem {
  id: int
  src: string
  type: 'image' | 'video' | 'audio'
  title: string
  mime: string
  duration: int | null
  poster: string           // only for video items; WP attachment image URL
}
```

## State Transitions

```
bio_parse:  raw_biography_string â†’ PlaylistPayload
item:       attachment_id â†’ MediaItem (or skipped if URL not resolvable)
```

## Conditional Logic

- `[gallery ids="..."]` â†’ items with `type = 'image'`; mime must match `image/*` or item skipped
- `[playlist ids="..." type="video"]` â†’ `type = 'video'`
- `[playlist ids="..." type="audio"]` â†’ `type = 'audio'`; default type for `[playlist]` without explicit type
- `duration` populated only if `wp_get_attachment_metadata($id)['length']` exists
- `poster` populated via `wp_get_attachment_image_url($id, 'large')` for video items only
- `title`: for images â€” `_wp_attachment_image_alt` meta; fallback `wp_basename($src)`; for audio/video â€” `get_the_title($id)`; fallback `wp_basename($src)`
- If `dokan_get_store_info` function not available: biography sourced from user meta only
- If biography is empty or not parseable: returns `{ items: [], fallbackImage: '', fallbackVideo: '' }`

None â€” read-only.


## Compatibility Adapter Behavior

- Uses `dokan_get_store_info()` as primary biography source
- Uses `dokan()->vendor->get($vendor_id)` for banner image resolution; falls back to user meta `dokan_banner`
- `dokan_banner_video` user meta for fallback video URL
- All WP attachment API calls unchanged between adapters

## Default WP Adapter Behavior

- **Biography source**: vendor profile CPT (`tm_vendor`) post content or custom post meta field (replaces `dokan_get_store_info`)
- **Banner video**: custom post meta `tm_banner_video` on vendor CPT (replaces `dokan_banner_video` user meta)
- **Banner image**: vendor CPT featured image â€” `get_post_thumbnail_id($vendor_cpt_id)` â†’ `wp_get_attachment_url` (replaces Dokan vendor model)
- Core parsing algorithm (shortcode regex + media resolution loop) unchanged between adapters
