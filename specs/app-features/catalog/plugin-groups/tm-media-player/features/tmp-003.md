# tmp-003 — REST Endpoint for Vendor Store Content Payload

## Business Objective

Expose a REST endpoint that returns the complete media player payload for a given vendor, enabling the AJAX navigation layer to fetch and hydrate the store page without a full reload.

## Actors

- **Browser client** — issues GET request on AJAX store navigation
- **System** — resolves vendor, assembles payload, responds with JSON

## UX Journey / User Actions

1. User navigates to a vendor store page via AJAX (e.g., clicking a vendor card in the store listing)
2. Client-side JS requests `GET /wp-json/tm-media-player/v1/store-content?vendor_id=N`
3. Server validates `vendor_id` parameter
4. Server calls `tm_get_vendor_media_playlist($vendor_id)` (tmp-001 contract)
5. Optionally: server buffers and returns rendered store section HTML fragments
6. Server returns JSON response: `{ vendor_id, playlist, sections? }`
7. Client injects playlist into player; sections into DOM regions

## Inputs

| Field | Type | Source |
|---|---|---|
| vendor_id | int | REST query parameter (`?vendor_id=N`) |
| sections[] | string[] optional | REST query parameter (which template sections to return) |

## Outputs

```
StoreContentResponse {
  vendor_id: int
  playlist: PlaylistPayload        // from tmp-001 contract
  sections: { [key: string]: string }   // optional: rendered HTML fragments per section name
  error: null | string
}
```

## State Transitions

None — read-only endpoint.

## Conditional Logic

- `vendor_id = 0` or non-existent user → `400` or `404` response with `error: 'invalid_vendor_id'` or `error: 'vendor_not_found'`
- `sections` param absent → return only `playlist`; no template buffering
- Template section inclusion only available in compatibility adapter (requires Dokan template paths)
- Authenticated vs non-authenticated: endpoint is **public** (no auth required for public vendor store content)

## Side Effects

None — read-only.

## Current Dependencies

| Dependency | Usage |
|---|---|
| `register_rest_route` | Endpoint registration |
| `tm_get_vendor_media_playlist()` | Playlist payload assembly (tmp-001) |
| Dokan store template paths | Template section HTML buffering (compatibility adapter) |
| WP REST controller infrastructure | `WP_REST_Request`, `WP_REST_Response`, `WP_Error` |

## Core Contract Definition

```
GET /wp-json/tm-media-player/v1/store-content

Query params:
  vendor_id (int, required)
  sections  (string[], optional)

Success (200):
  {
    vendor_id: int,
    playlist:  PlaylistPayload,
    sections?: { [section_name]: html_string }
  }

Error responses:
  400: { error: 'invalid_vendor_id' }   // vendor_id <= 0
  404: { error: 'vendor_not_found' }    // vendor_id > 0 but user does not exist
```

## Compatibility Adapter Behavior

- Registered via `register_rest_route('tm-media-player/v1', '/store-content', ...)`
- Playlist resolved via `tm_get_vendor_media_playlist($vendor_id)`
- Template sections loaded by buffering Dokan template file includes via `ob_start()` + `require`
- Dokan template path resolution uses Dokan's theme template hierarchy

## Default WP Adapter Behavior

- Same REST endpoint registration and structure
- Playlist resolved via default WP adapter version of `extractPlaylist()` (tmp-001 default adapter)  
- Template sections from WP block-rendered template parts (no Dokan theme paths)
- Vendor existence check via `get_post()` on `tm_vendor` CPT (replaces `get_user_by('id', ...)`)

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Payload structure | Same JSON schema for `playlist` field for equivalent vendor state |
| 200 | Valid vendor_id with media content returns 200 |
| 404 | Non-existent vendor_id returns 404 with `error: 'vendor_not_found'` |
| 400 | `vendor_id = 0` or absent returns 400 with `error: 'invalid_vendor_id'` |
| Public access | No auth required; endpoint accessible without cookie or token |
