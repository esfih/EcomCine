# tap-002 — AJAX Login and Session Bootstrap

## Business Objective

Authenticate a user via a front-end AJAX request and return session state to the client, enabling modal-based login without page reload.

## Actors

- **Guest visitor** — submits credentials via account panel modal
- **System** — validates nonce, resolves identifier, authenticates, returns result

## UX Journey / User Actions

1. Guest clicks account tab → modal opens in login mode (tap-001)
2. User enters username/email and password → submits form
3. JS sends `POST` to `wp-admin/admin-ajax.php` action `tm_account_login_user`
4. Server checks `check_ajax_referer('tm_account_login', 'nonce')` → `403` on failure
5. Server resolves identifier: `@` present → `get_user_by('email', ...)` → extract `user_login`; else treat as username
6. Server calls `wp_signon(['user_login' => ..., 'user_password' => ..., 'remember' => true])`
7. On WP_User returned: `wp_set_current_user($user->ID)`; `wp_set_auth_cookie($user->ID, true)`; respond `{ success: true }`
8. On WP_Error: respond `{ success: false, message: error_message }`
9. JS receives response → on success: closes modal, triggers account panel update; on failure: shows inline error

## Inputs

| Field | Type | Source |
|---|---|---|
| `identifier` | string (username or email) | POST body |
| `password` | string | POST body |
| `nonce` | string | POST body (key: `nonce`) |
| AJAX action | `tm_account_login_user` | POST body (action key) |

## Outputs

| Output | Type | Condition |
|---|---|---|
| `{ success: true }` | JSON | Authentication succeeded |
| `{ success: false, message: string }` | JSON | Authentication failed |
| WP auth cookie | HTTP cookie | Authentication succeeded |

## State Transitions

```
auth_state:  unauthenticated → authenticated     (on success)
session:     none → established                  (WP auth cookie set)
modal_state: open(login_form) → closed           (JS, on success)
             open(login_form) → open(error_shown) (JS, on failure)
```

## Conditional Logic

- Nonce check fails → `wp_send_json_error(['message' => 'Invalid nonce'])` (no auth attempt)
- Identifier contains `@` → resolve as email: `get_user_by('email', $identifier)` → use `user_login`; if user not found → `WP_Error`
- Identifier without `@` → use directly as `user_login`
- `wp_signon` returns `WP_Error` → extract and return sanitized error message
- `remember = true` for persistent auth cookie

## Side Effects

- WP authentication cookie set on success (`wp_set_auth_cookie`)
- `wp_login` action fired by `wp_signon`
- Login timestamp stored in WP user meta (WP core behavior)

## Current Dependencies

| Dependency | Usage |
|---|---|
| `wp_ajax_nopriv_tm_account_login_user` | AJAX endpoint for unauthenticated users |
| `wp_ajax_nopriv_tm_account_ping` | Session state ping for unauthenticated users |
| `check_ajax_referer` | Nonce verification |
| `get_user_by('email', ...)` | Email → username resolution |
| `wp_signon` | Authentication |
| `wp_set_current_user` | Sets authenticated user in WP context |
| `wp_set_auth_cookie` | Sets browser auth cookie |
| `wp_send_json_success` / `wp_send_json_error` | JSON response helpers |

## Core Contract Definition

```
login(identifier: string, password: string, nonce: string) → LoginResult

LoginResult {
  success: true
  | { success: false, message: string }
}

// Identifier can be email or username; system normalizes to username before wp_signon
// Nonce must match 'tm_account_login' or request rejected (HTTP 403 equivalent)
// Side effect on success: WP auth cookie established in response
```

## Compatibility Adapter Behavior

- `admin-ajax.php` endpoint with `wp_ajax_nopriv_tm_account_login_user` action
- `check_ajax_referer('tm_account_login', 'nonce')` for request validation
- `wp_signon` + `wp_set_auth_cookie` for session establishment
- Error messages passed through from `WP_Error::get_error_message()`

## Default WP Adapter Behavior

- WP REST endpoint: `POST /wp-json/tm/v1/auth/login` (replaces admin-ajax)
- Same nonce validation pattern (`X-WP-Nonce` header or body nonce)
- Same `wp_signon` + `wp_set_auth_cookie` calls (unchanged WP auth mechanism)
- REST-compatible JSON response format
- `wp_ajax_nopriv` actions no longer needed

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Valid credentials | Returns `{ success: true }` and sets auth cookie |
| Invalid password | Returns `{ success: false, message: 'incorrect_password' equivalent }` |
| Invalid username | Returns `{ success: false, message: 'invalid_username' equivalent }` |
| Invalid nonce | Request rejected before auth attempt |
| Email identifier | Email resolves to correct user login; auth succeeds with correct password |
| Session establishment | Auth cookie present in browser after success |
