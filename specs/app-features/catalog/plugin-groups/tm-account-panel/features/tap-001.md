# tap-001 — Store-Page Account Panel Shell and Modal UX

## Business Objective

Render a right-side account panel tab on supported store/listing/platform pages, providing an entry point to the login/register modal and account state view accessible from the vendor showcase surface.

## Actors

- **Guest visitor** — sees login/register modal on panel open
- **Logged-in user** — sees their account state sections in the panel
- **System** — detects eligible pages; enqueues assets; localizes JS data

## UX Journey / User Actions

1. User visits a vendor store page, store listing, talent showcase, or platform page
2. Plugin runs `tm_account_panel_is_store_page()` to determine eligibility
3. If eligible: `wp_enqueue_style` + `wp_enqueue_script` for `account-panel.css`/`account-panel.js`
4. `wp_localize_script` injects `tmAccountPanel` object (ajaxUrl, nonces, privacy/terms URLs, debug flag)
5. JS renders the account panel tab/trigger in the page layout
6. User clicks tab → modal opens; shows login form (guest) or account sections (authenticated)
7. On successful login: modal updates to account state (auth_state transitions — see tap-002)

## Inputs

| Field | Type | Source |
|---|---|---|
| Current page context | bool from `tm_account_panel_is_store_page()` | WP hooks |
| is_admin() | bool | WP global |
| Auth state | is_user_logged_in() | WP session |

## Outputs

| Output | Description |
|---|---|
| Enqueued CSS | `tm-account-panel-css` stylesheet on eligible pages |
| Enqueued JS | `tm-account-panel-js` script on eligible pages |
| Localized JS | `tmAccountPanel` object: ajaxUrl, nonces, URLs, debug |
| Account panel HTML | Rendered by JS into page DOM on load |

## State Transitions

```
panel_visibility:  hidden → visible     (eligible page detected)
                   visible → hidden     (non-eligible page)
modal_state:       closed → open → closed
auth_state:        guest → authenticated (after login, tap-002)
                   authenticated → account_panel_shown
```

## Conditional Logic

- Page eligibility: `tm_is_showcase_page() || dokan_is_store_page() || dokan_is_store_listing() || is_page_template('page-platform.php')` → `true` → enqueue assets
- `is_admin()` check: assets never enqueued in admin context
- If `get_privacy_policy_url()` returns empty: fallback to `{home_url}/privacy`
- Panel sections (`orders`, `bookings`, `IP`) each gated by role/capability checks in tap-004

## Side Effects

- CSS/JS enqueued on eligible pages (performance impact on page weight)
- Nonces generated at enqueue time (`tm_account_login`, `tm_account_admin`, `tm_account_orders`)
- JS debug mode enabled if `WP_DEBUG` is true

## Current Dependencies

| Dependency | Usage |
|---|---|
| `dokan_is_store_page()` | Store page detection |
| `dokan_is_store_listing()` | Store listing detection |
| `tm_is_showcase_page()` | Platform showcase page detection |
| `is_page_template('page-platform.php')` | Platform page template detection |
| `wp_enqueue_script` / `wp_enqueue_style` | Asset loading |
| `wp_localize_script` | Inject JS config data |
| `wp_create_nonce` | Generate multiple action nonces |

## Core Contract Definition

```
isEligiblePage() → bool
  // true if: showcase page OR Dokan store page OR Dokan store listing OR platform page template

enqueueAccountPanel(page_context, auth_state) → void
  // Enqueues CSS + JS for account panel
  // Localizes tmAccountPanel: { ajaxUrl, nonce, adminNonce, orderNonce,
  //                             privacyUrl, talentTermsUrl, hirerTermsUrl, debug }

renderAccountPanel(auth_state: 'guest' | 'authenticated') → DOM mutation (JS)
  // Guest: show login/register form
  // Authenticated: show account sections (tap-004)
```

## Compatibility Adapter Behavior

- `isEligiblePage()` uses `dokan_is_store_page()` and `dokan_is_store_listing()` for Dokan context
- `is_page_template('page-platform.php')` for platform page
- `tm_is_showcase_page()` as non-Dokan override

## Default WP Adapter Behavior

- `isEligiblePage()` uses `is_author() || is_post_type_archive('tm_vendor') || is_page_template('page-platform.php')`
- Eliminates `dokan_is_store_page()` and `dokan_is_store_listing()` entirely
- Same `wp_enqueue_*` and `wp_localize_script` calls; no Dokan dependency in asset decision

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Panel visible on store page | Panel renders on Dokan store and equivalent WP vendor page |
| Panel visible on listing | Panel renders on store listing and equivalent vendor archive |
| Panel absent elsewhere | Panel absent on WP shop page, product single, blog, etc. |
| Modal opens on tap | Account modal opens on panel tab click |
| Auth transitions | Login updates panel to account state (tap-002 outcome visible) |
