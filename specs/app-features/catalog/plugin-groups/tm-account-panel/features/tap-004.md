# tap-004 — Vendor Account Management Panel (Orders, Bookings, IP Sections)

## Business Objective

Present a logged-in vendor/talent user with a contextualized account dashboard showing their orders, bookings, and IP assets (invitations/shares) in a panel accessible from the account tab.

## Actors

- **Logged-in vendor/talent** — views their orders, bookings, and IP assets
- **System** — fetches and renders each section via AJAX on tab activation

## UX Journey / User Actions

1. Logged-in user is on an eligible page (tap-001); account panel shows account state
2. Panel has section tabs: **Orders**, **Bookings**, **IP**
3. User activates a tab → JS sends AJAX request for that section's data
4. **Orders**: AJAX fires `tm_account_orders` action → returns WooCommerce orders for current user
5. **Bookings**: AJAX fires `tm_account_bookings` action → returns WooCommerce Bookings for current user
6. **IP** section: AJAX fires `tm_account_ip` action → returns invitation tokens and share links from user meta/custom table
7. Response data injected into tab content area as rendered HTML or JSON for JS rendering
8. Each section independently refreshable

## Inputs

| Field | Type | Source |
|---|---|---|
| user_id | int | WP session (current_user) |
| section | `orders` \| `bookings` \| `ip` | AJAX param |
| nonce | string | Section-specific nonce (orderNonce, etc.) |
| page | int optional | Pagination parameter |

## Outputs

```
OrdersSection {
  orders: [{
    order_id: int,
    date: string,
    status: string,
    total: string,
    vendor_name: string
  }]
}

BookingsSection {
  bookings: [{
    booking_id: int,
    product_name: string,
    date: string,
    status: string,
    vendor_name: string
  }]
}

IPSection {
  invitations: [{
    token: string,
    talent_name: string,
    lifecycle_stage: string,
    invited_at: string,
    claimed_at: string|null
  }],
  shares: [{
    share_url: string,
    talent_name: string,
    expiry: string
  }]
}
```

## State Transitions

```
section_state:  unloaded → loading → loaded → refreshing → loaded
```

## Conditional Logic

- **Orders section** visibility: user has at least one WooCommerce order as customer
- **Bookings section** visibility: user has at least one WooCommerce Booking
- **IP section** visibility: user has at least one invitation record linked (admin-linked or self-invited)
- Each section request must pass its own nonce check before data is returned
- Pagination supported for orders/bookings; default page size configurable

## Side Effects

None — read-only.

## Current Dependencies

| Dependency | Usage |
|---|---|
| WooCommerce `wc_get_orders(['customer' => $user_id])` | Fetch orders for current user |
| WooCommerce Bookings `WC_Booking_Data_Store::get_customer_bookings($user_id)` | Fetch bookings |
| Dokan earnings API | Vendor earnings data (if shown in panel) |
| `wp_ajax_tm_account_orders` | AJAX action for orders |
| `wp_ajax_tm_account_bookings` | AJAX action for bookings |
| `wp_ajax_tm_account_ip` | AJAX action for IP assets |
| `get_user_meta` | Invitation/share data (from tap-003 storage) |

## Core Contract Definition

```
getOrders(user_id: int, page: int = 1) → OrdersSection
getBookings(user_id: int, page: int = 1) → BookingsSection
getIPAssets(user_id: int) → IPSection

// All require valid nonce for their respective action
// All reject unauthenticated requests
// Empty arrays returned if user has no items (not an error)
```

## Compatibility Adapter Behavior

- `wc_get_orders(['customer' => $user_id, 'limit' => 10, 'offset' => ...])` for orders
- `WC_Booking_Data_Store` for bookings
- Dokan earnings API for vendor-specific earned amount
- `wp_ajax_*` handlers in `tm-account-panel.php`

## Default WP Adapter Behavior

- **Orders**: custom CPT `tm_order` queried via `WP_Query` (post_author = user_id) — eliminates WooCommerce dependency
- **Bookings**: custom CPT `tm_booking` queried via `WP_Query` — eliminates WooCommerce Bookings dependency
- **IP assets**: `tm_invitation` CPT queried (from tap-003 default adapter)
- All AJAX actions replaced with WP REST endpoints:
  - `GET /wp-json/tm/v1/account/orders`
  - `GET /wp-json/tm/v1/account/bookings`
  - `GET /wp-json/tm/v1/account/ip-assets`

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Orders section | Same order records visible for equivalent WC order state |
| Bookings section | Same booking records visible for equivalent WC Bookings state |
| IP section | Same invitation records for equivalent tap-003 state |
| Empty sections | Empty arrays returned gracefully when user has no records |
| Section visibility gates | Section absent from panel when user has no relevant records |
| Auth enforcement | Unauthenticated requests rejected for all sections |
