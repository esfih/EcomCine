# tap-003 — Admin-Assisted Talent Onboarding and Claim/Share Flows

## Business Objective

Enable admins to onboard talent users through invitation, claiming, and sharing workflows, transitioning users through pre-defined lifecycle stages and assigning appropriate WP roles.

## Actors

- **Admin** — initiates onboarding, generates invitation tokens, monitors lifecycle
- **Talent user** — receives and accepts an invitation/claim link
- **System** — generates tokens, manages lifecycle transitions, assigns roles

## UX Journey / User Actions

1. Admin opens account panel IP section (or admin page) → selects a talent user
2. Admin triggers **Create Invitation** → system generates a unique token + expiry
3. System stores token in WP user meta (or custom table) linked to talent user
4. Admin receives a `claim_url` containing the token, shares it with the talent
5. Talent visits `claim_url` → system validates token (existence + expiry + not already claimed)
6. On valid token: system transitions talent to `claimed` lifecycle stage; assigns WP role
7. Admin views claim status (claimed/unclaimed/expired) in IP section
8. **Share flow**: admin generates a short-lived share link for talent profile (non-claim token)

## Inputs

| Field | Type | Source |
|---|---|---|
| talent_user_id | int | Admin UI selection |
| action_type | `invite` \| `claim` \| `share` | Admin UI or URL |
| token | string | Claim URL query param |
| admin_nonce | string | `tm_account_admin` action nonce |

## Outputs

| Output | Description |
|---|---|
| Invitation token | Unique string stored as user meta; returned to admin client |
| `claim_url` | Full URL containing token (`?tm_claim={token}`) |
| Lifecycle transition | WP user role updated to `talent` or `claimed_talent` |
| JSON response | `{ success, token?, claim_url?, error? }` to admin client |

## State Transitions

```
talent_lifecycle:   anonymous → invited          (admin creates invitation)
                    invited   → claimed           (talent visits claim URL with valid token)
                    invited   → expired           (token past expiry without claim)
                    claimed   → active            (admin or system confirms activation)
token_state:        generated → consumed          (on successful claim)
                    generated → expired           (past expiry_datetime)
```

## Conditional Logic

- Admin capability check (`current_user_can('manage_options')` or custom cap) required for token generation
- Token expiry enforced: `expiry_datetime < now()` → token invalid, claim rejected
- Token already consumed → claim rejected with `error: 'already_claimed'`
- Role assignment based on lifecycle stage: `claimed` → add WP role `talent`
- Share tokens: shorter expiry, no role transition; only used for profile link sharing

## Side Effects

- WP user role updated via `wp_update_user` or `$user->add_role('talent')`
- Token stored in WP user meta (key: `tm_invitation_token`, `tm_invitation_expiry`)
- Email notification sent to talent (via `wp_mail`) if email trigger configured
- Audit log entry (user meta `tm_claimed_at` timestamp) on successful claim

## Current Dependencies

| Dependency | Usage |
|---|---|
| `wp_ajax_tm_account_admin_*` | AJAX actions for admin-initiated flows |
| `wp_ajax_nopriv_tm_account_claim` | Public claim URL handler |
| `wp_generate_password(32, false)` | Secure token generation |
| `wp_update_user` / `$user->add_role` | Role assignment |
| `get_user_meta` / `update_user_meta` | Token storage and lifecycle tracking |
| `wp_mail` (optional) | Invitation email delivery |

## Core Contract Definition

```
createInvitation(talent_user_id: int, admin_nonce: string) → InvitationResult
  { token: string, expiry: datetime, claim_url: string }
  | { error: string }

claimInvitation(token: string) → ClaimResult
  { success: true, user_id: int, role: string }
  | { error: 'invalid_token' | 'expired_token' | 'already_claimed' }

getOnboardingStatus(talent_user_id: int) → OnboardingStatus
  { lifecycle_stage: 'anonymous' | 'invited' | 'claimed' | 'active',
    invited_at?: datetime,
    claimed_at?: datetime,
    token_expiry?: datetime }

createShareLink(talent_user_id: int) → ShareResult
  { share_url: string, expiry: datetime }
```

## Compatibility Adapter Behavior

- Admin-post / `wp_ajax_*` handlers in `tm-account-panel.php`
- Tokens stored as WP user meta (simple key-value, no custom table)
- Role transitions via WP `wp_update_user` / `add_role`
- `admin-ajax.php` as the transport layer

## Default WP Adapter Behavior

- REST endpoints:
  - `POST /wp-json/tm/v1/invitations` → create invitation
  - `GET /wp-json/tm/v1/invitations/{token}` → validate and claim
  - `GET /wp-json/tm/v1/invitations/status/{user_id}` → lifecycle status
- Custom post type `tm_invitation` for token lifecycle (replaces user meta storage; enables richer querying)
- Same WP role management (`add_role` / `wp_update_user`) unchanged

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Lifecycle stages | Same lifecycle transitions for equivalent admin actions |
| Token validation | Valid token → claim success; expired/consumed token → correct error |
| Role assignment | Talent role added on successful claim |
| Admin restriction | Non-admin cannot generate invitation tokens |
| Share links | Non-claim share URL generated with correct expiry; no role change on visit |
