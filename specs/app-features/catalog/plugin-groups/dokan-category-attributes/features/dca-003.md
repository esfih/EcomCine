# dca-003 — Storefront Vendor Profile Attribute Display

## Business Objective

Render a vendor's category-specific attribute values publicly on their Dokan store profile page in a structured, styled format — determined by category membership and individual field visibility flags.

## Actors

- **Buyer/Visitor** — views attribute data on the vendor store page
- **Vendor** — data source (attribute values populated via dca-002 dashboard form)
- **System** — resolves, filters, and renders attributes at hook injection point

## UX Journey / User Actions

1. Buyer opens a vendor's Dokan store page
2. Plugin hooks `dokan_store_profile_frame_after` (priority 5) — executes early in Dokan's store frame
3. System resolves `$store_user->ID` → queries vendor's `store_category` terms
4. Iterates active attribute sets; for each: checks if `array_intersect($set->categories, $vendor_categories) > 0`
5. For matching sets: queries fields where `show_in_public = 1`
6. For each public field: reads `get_user_meta($vendor_id, $field->field_name, true)`
7. Skips fields with empty values; if at least one value exists → renders the attribute set block
8. Outputs within `.vendor-custom-attributes-wrapper`

## Inputs

| Field | Type | Source |
|---|---|---|
| `$store_user` | WP_User | Dokan hook arg |
| `$store_info` | array | Dokan hook arg |
| vendor's `store_category` terms | taxonomy slugs array | `wp_get_object_terms` |

## Outputs

| Output | Description |
|---|---|
| HTML block | `.vendor-custom-attributes-wrapper` containing attribute set sections with field/value pairs |
| Empty output | Rendered if vendor has no matching categories or no populated public fields |

## State Transitions

```
display:  nothing → rendered    (at least one field with value exists for matching category)
          nothing → nothing     (no applicable sets or all values empty)
```

## Conditional Logic

- Attribute set rendered only if `array_intersect($set->categories, $vendor_categories)` is non-empty
- Only fields with `show_in_public = 1` included
- Only fields where `get_user_meta($vendor_id, $field->field_name, true)` is non-empty rendered
- If vendor has no `store_category` terms: no categories matched → no output
- Each attribute set block rendered independently; empty sets suppressed

## Side Effects

None — read-only display operation.

## Current Dependencies

| Dependency | Usage |
|---|---|
| Dokan action `dokan_store_profile_frame_after` | Injection point on public store page |
| Dokan concept: store page context | Provides `$store_user` and `$store_info` to hook callback |
| `store_category` taxonomy | Category-scope filter via `wp_get_object_terms` |
| `get_user_meta` | Reads vendor attribute values (key = `field_name`) |
| `DCA_Attribute_Manager::get_attribute_sets()` | Fetches active sets |
| `DCA_Attribute_Manager::get_fields()` | Fetches public fields per set |

## Core Contract Definition

```
projectVendorAttributes(vendor_id) → AttributeSetProjection[]

AttributeSetProjection {
  set_name: string
  set_icon: string|null
  fields: AttributeFieldProjection[]
}

AttributeFieldProjection {
  label: string
  icon: string|null
  value: string|array   // formatted per field_type
}

// Rules:
// - Only attribute sets matching vendor's store_category terms included
// - Only fields where show_in_public = 1 included
// - Only fields where value is non-empty included
// - Sets with zero populated public fields excluded
```

## Compatibility Adapter Behavior

- Hooks `dokan_store_profile_frame_after` to inject output into Dokan store page template
- Relies on Dokan to supply `$store_user` (vendor WP_User) via hook argument
- Vendor category resolved via `wp_get_object_terms($vendor_id, 'store_category')`
- Output is plain PHP rendered HTML (no AJAX on public side)

## Default WP Adapter Behavior

- WP block render callback on vendor profile CPT (post_type=`tm_vendor`) or author archive template
- REST endpoint: `GET /wp-json/dca/v1/vendor-attributes/{vendor_id}/public` → `AttributeSetProjection[]`
- Template part or block consuming the REST response or calling the core projection function directly
- No Dokan hook dependency; template injection via block-theme template part

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Category-scoped rendering | Correct attribute sets shown/hidden per category membership |
| Empty field suppression | Fields with no value absent from rendered output |
| Value accuracy | Displayed values match stored user meta for equivalent state |
| Set ordering | Sets rendered in ascending `priority` order |
| Feature flag | `show_in_public = 0` fields never appear on public page |
