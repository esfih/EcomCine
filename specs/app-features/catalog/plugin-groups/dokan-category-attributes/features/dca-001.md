# dca-001 — Attribute Schema and Storage Tables

## Business Objective

Provide a category-scoped attribute schema registry allowing admins to define typed field sets per vendor category, and persist vendor-supplied attribute values for query and display.

## Actors

- **Admin** — creates and manages attribute sets and field definitions via WP admin
- **System** — enforces schema on save/read; resolves category scope for each request

## UX Journey / User Actions

1. Admin opens WP Admin → Dokan Category Attributes → Add Attribute Set
2. Admin assigns name, slug, icon, category slugs (JSON array), and sort priority
3. Admin adds fields to the set: each field has name, label, type, options, display visibility flags
4. System persists schema to `{prefix}dokan_attribute_sets` and `{prefix}dokan_attribute_fields`
5. Vendor submits profile form → attribute values stored in WP user meta (field_name as meta key)
6. Optional: vendor attribute values mirrored to `{prefix}dokan_vendor_attributes` for richer querying

## Inputs

| Field | Type | Source |
|---|---|---|
| attribute_set: name | string | Admin form |
| attribute_set: slug | string | Admin form (unique) |
| attribute_set: icon | string | Admin form (dashicons slug) |
| attribute_set: categories | JSON array of slugs | Admin form |
| attribute_set: priority | int | Admin form (default 10) |
| attribute_set: status | active \| inactive | Admin form |
| attribute_field: field_name | string | Admin form (unique) |
| attribute_field: field_label | string | Admin form |
| attribute_field: field_type | text \| select \| checkbox \| textarea \| number | Admin form |
| attribute_field: field_options | JSON key-value | Admin form |
| attribute_field: required | bool | Admin form |
| attribute_field: show_in_dashboard | bool | Admin form |
| attribute_field: show_in_public | bool | Admin form |
| attribute_field: show_in_filters | bool | Admin form |

## Outputs

| Output | Description |
|---|---|
| Attribute set records | Rows in `{prefix}dokan_attribute_sets` |
| Attribute field records | Rows in `{prefix}dokan_attribute_fields` |
| Vendor attribute values | WP user meta entries (meta_key = field_name) |
| Schema version option | `dca_db_version` in wp_options |

## State Transitions

```
attribute_set:   created(active) → updated → deactivated(inactive) → deleted(cascade to fields)
attribute_field: created → updated → deleted (cascade on set delete)
vendor_value:    empty → populated → updated
```

## Conditional Logic

- A field is displayed or queried **only** if its parent set's categories array intersects with the vendor's assigned `store_category` taxonomy terms
- Fields with `required = 1` must be non-empty for the set to be considered fully populated
- `show_in_dashboard`, `show_in_public`, and `show_in_filters` independently gate each field per rendering context
- Schema is applied at plugin activation via `dbDelta()`; migrations guarded by `dca_db_version` option

## Side Effects

- `dbDelta()` executed on plugin activation → creates/updates three custom tables
- `update_option('dca_db_version', DCA_VERSION)` on activation
- `DELETE option + DROP TABLE` on plugin uninstall (irreversible)

## Current Dependencies

| Dependency | Usage |
|---|---|
| `{prefix}dokan_attribute_sets` | Custom DB table (managed by `DCA_Database`) |
| `{prefix}dokan_attribute_fields` | Custom DB table (managed by `DCA_Database`) |
| `{prefix}dokan_vendor_attributes` | Optional custom table for structured vendor value queries |
| Dokan `store_category` taxonomy | Category-scope visibility resolution |
| WP user meta (`wp_usermeta`) | Primary storage for vendor attribute values |
| `DCA_Attribute_Manager` | Business logic layer for schema reads and export/import |

## Core Contract Definition

```
createAttributeSet(name, slug, categories[], priority, status) → set_id
readAttributeSets(filters?) → AttributeSet[]
updateAttributeSet(set_id, data) → bool
deleteAttributeSet(set_id) → bool  // cascades to fields and vendor values

createField(set_id, field_config) → field_id
readFields(set_id, location?: dashboard|public|filters) → Field[]
updateField(field_id, data) → bool
deleteField(field_id) → bool

writeVendorValue(vendor_id, field_name, value) → bool
readVendorValue(vendor_id, field_name) → string|array|null
getFieldsForVendor(vendor_id) → Field[]  // scoped by category intersection

exportAttributeSet(set_id) → JSON string
importAttributeSet(json) → set_id | WP_Error
```

## Compatibility Adapter Behavior

- Attribute sets and fields stored in custom DB tables via `DCA_Database` class
- Vendor attribute values stored in WP user meta (`get_user_meta` / `update_user_meta`, key = `field_name`)
- Vendor category resolution via `wp_get_object_terms($vendor_id, 'store_category', ['fields' => 'slugs'])`
- Admin interface via `DCA_Admin_Menu` (WP admin sub-menu page under Dokan)
- Export/import via `wp_ajax_dca_export_set` / `wp_ajax_dca_import_set` with nonce and capability checks

## Default WP Adapter Behavior

- **Attribute set schema** → WP custom post type `dca_attribute_set`; category associations via taxonomy relationships on the CPT
- **Attribute fields** → child CPT posts or post meta within the set CPT (no raw SQL)
- **Vendor attribute values** → WP user meta (field_name key) — already WP-native; no change
- **Category scope** → WP_Query `tax_query` on `store_category` (or a WP-native equivalent vendor category taxonomy)
- Admin editing via native WP CPT edit screen with custom meta boxes (no custom DB admin page)

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Field set scoping | Same field set returned for a given vendor based on category intersection |
| Value roundtrip | Read-after-write returns identical value for all field types |
| Required enforcement | Save blocked or flagged when required fields are empty |
| Export/import cycle | Full JSON roundtrip produces equivalent schema with no data loss |
| Schema version | `dca_db_version` present after activation; tables exist |
