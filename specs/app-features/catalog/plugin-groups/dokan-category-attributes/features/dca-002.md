# dca-002 — Vendor Dashboard Dynamic Field Rendering

## Business Objective

Inject category-appropriate attribute form fields into the Dokan vendor profile/settings dashboard, with real-time show/hide based on the vendor's active category selection.

## Actors

- **Vendor** — views and fills attribute fields within their Dokan dashboard
- **System** — injects fields via action hook; persists values on profile save

## UX Journey / User Actions

1. Vendor opens Dokan Dashboard → Settings → Store Settings
2. Fields for the vendor's current category display immediately; non-matching sets are hidden (`display:none`)
3. Vendor selects/changes a category → JS reads `data-category` attributes and shows/hides relevant attribute sections in real time
4. Vendor fills in fields: `select`, `checkbox`, `textarea`, `text`, or `number` inputs
5. Vendor clicks **Save** → Dokan fires `dokan_store_profile_saved` action
6. Plugin handler iterates fields, sanitizes per type, and persists via `update_user_meta`

## Inputs

| Field | Type | Source |
|---|---|---|
| store_id | int | Dokan hook arg |
| store_settings | array | Dokan hook arg |
| category selection | store_category slugs → `wp_get_object_terms` result | Vendor profile |
| field values | POST payload keyed by `field_name` | Vendor form submission |

## Outputs

| Output | Description |
|---|---|
| HTML form sections | One `<div class="dca-attribute-section">` per applicable attribute set |
| Persisted meta | `update_user_meta($vendor_id, $field->field_name, value)` per field on save |

## State Transitions

```
field_group:  hidden → visible    (category match detected on page load or JS change)
              visible → hidden     (category deselected)
field_value:  empty → draft        (vendor types value, not yet saved)
              draft → saved        (profile form submitted successfully)
```

## Conditional Logic

- `array_intersect($set->categories, $vendor_categories) > 0` → show the attribute section; else `display:none`
- Each individual field is also subject to `field->show_in_dashboard = 1` (fields with `0` are never rendered in dashboard)
- `required = 1` fields emit a browser-level `required` attribute; server-side required validation on save
- Category visibility re-evaluated via JavaScript on the Dokan category-change event (reads `data-category` on each wrapper div)

## Side Effects

- `update_user_meta($vendor_id, $field->field_name, $sanitized_value)` per saved field
- Profile completeness score may re-evaluate post-save (if tho-005 completeness module is active)

## Current Dependencies

| Dependency | Usage |
|---|---|
| Dokan action `dokan_settings_after_store_phone` | Injection point for field HTML |
| Dokan action `dokan_store_profile_saved` | Save trigger |
| `wp_get_object_terms($store_id, 'store_category')` | Vendor category resolution |
| `get_user_meta` / `update_user_meta` | Field value read/write (key = `field_name`) |
| `DCA_Attribute_Manager` | Resolves active attribute sets and fields |

## Core Contract Definition

```
renderDashboardFields(vendor_id) → HTML
  // Returns one section wrapper per matching attribute set
  // Each section: data-category, initial display state, field inputs

persistFieldValues(vendor_id, POST_payload) → {saved: string[], errors: string[]}
  // Validates each field value by field_type
  // Sanitizes: text → sanitize_text_field; textarea → sanitize_textarea_field;
  //            select → enum check; checkbox → array values check
  // Calls update_user_meta for each valid field
```

## Compatibility Adapter Behavior

- Hooks `dokan_settings_after_store_phone` to inject HTML at the correct position in Dokan's store settings form
- Hooks `dokan_store_profile_saved` to intercept save and iterate field values from `$_POST`
- JS-driven category-conditional show/hide reads `data-category` from each section wrapper `div`
- Category detection uses Dokan's `store_category` taxonomy via `wp_get_object_terms`

## Default WP Adapter Behavior

- Gutenberg sidebar panel block (or `add_meta_box` equivalent) for vendor profile CPT edit screen
- REST endpoint `GET /wp-json/dca/v1/vendor-fields?vendor_id=N` returns field schema with current values
- REST endpoint `POST /wp-json/dca/v1/vendor-fields` accepts field payload, validates, and persists via user meta or CPT meta
- Block-based conditional visibility in vendor profile editor via block attributes; no Dokan action hooks needed

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Field set visibility | Same attribute sections visible when vendor's category matches set's assigned categories |
| Hidden sets | Non-matching category sets remain hidden |
| Validation gates | Required field blocks save until populated |
| Value persistence | All field types (text, select, checkbox, textarea, number) save and read back correctly |
| JS reactivity | Category change triggers correct show/hide without page reload |
