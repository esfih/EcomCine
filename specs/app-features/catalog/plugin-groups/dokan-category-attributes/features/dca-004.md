# dca-004 â€” Store Listing Filters by Attributes

## Business Objective

Render attribute-based filter controls on the Dokan store listing page and apply selected filter values to the vendor query, returning only vendors whose profile matches the selected attribute criteria.

## Actors

- **Buyer/Visitor** â€” selects filter values to narrow vendor listings
- **System** â€” renders filter UI and modifies vendor query arguments

## UX Journey / User Actions

1. Buyer opens Dokan store listing page (e.g., `/stores/`)
2. Plugin hooks `dokan_before_store_lists_filter_search` to inject filter control groups
3. All filter groups rendered with `display:none` initially
4. JS reads the active `store_category` selection and shows matching attribute set filter groups
5. Buyer selects a filter value (drop-down, checkbox, or text input)
6. Page reloads or form submits with GET params (e.g., `?talent_type=actor`)
7. Plugin hooks `dokan_seller_listing_args` to add `meta_query` clauses for active GET params
8. Dokan re-executes vendor query with modified args â†’ filtered vendor list rendered

## Inputs

| Field | Type | Source |
|---|---|---|
| Active store_category | taxonomy slug | GET param or JS-selected store_category filter |
| Filter params | `field_name â†’ value` pairs | GET request query string |
| Attribute field schema | set + fields with `show_in_filters = 1` | `DCA_Attribute_Manager` |

## Outputs

| Output | Description |
|---|---|
| Filter HTML | One `<div class="dca-filter-section">` per applicable attribute set, grouped per field |
| Modified query args | `dokan_seller_listing_args` with `meta_query` entries for each active filter param |

## State Transitions

```
filter_group:  hidden â†’ visible     (matching store_category selected via JS)
               visible â†’ hidden     (category deselected or changed)
vendor_query:  unfiltered â†’ filtered  (when GET params with known field_name keys are present)
               filtered â†’ unfiltered  (all filter params removed)
```

## Conditional Logic

- Filter groups rendered with `display:none`; JS shows groups whose `data-category` intersects with the active category selection
- Only fields with `show_in_filters = 1` rendered as filter controls
- For each active GET param matching a known `field_name`: `meta_query` clause appended:
  - `select` / `text`: `comparison = '='`
  - `checkbox` (array): `comparison = 'IN'` or `LIKE` per implementation
  - `number`: `comparison = '='` (exact) or range if min/max keys present
- Unknown GET params (not in field schema) ignored with no query modification

## Side Effects

None â€” display + query modification only; no writes.

## Compatibility Adapter Behavior

- Injects filter HTML via `dokan_before_store_lists_filter_search` action
- Reads `$_GET` for active filter values (sanitized before use)
- Applies `meta_query` modifications to args array in `dokan_seller_listing_args` filter
- JS show/hide on category change uses `data-category` attribute on filter section wrappers

## Default WP Adapter Behavior

- Block-based filter sidebar for vendor archive CPT list page (no Dokan action hook needed)
- REST endpoint: `GET /wp-json/dca/v1/vendors?[field_name]=[value]` â†’ paginated vendor list
- Server-side `WP_Query` / `WP_User_Query` with `meta_query` applied by REST controller
- React/block filter UI updates URL params and re-fetches via REST (replaces GET-reload pattern)

## Parity Oracle

| Assertion | Pass Condition |
|---|---|
| Filter controls visible | Same filter controls appear for the matching store_category |
| Filter controls hidden | Non-matching category filter groups remain hidden |
| Query accuracy | Same vendor set returned for equivalent attribute filter combinations |
| No false positives | Vendors without matching attribute values excluded |
| No false negatives | Vendors with matching attribute values included |
| Unknown params ignored | GET params not in field schema produce no query modification |
