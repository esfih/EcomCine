# tvt-001 — Browser-side WebM/VP9 Video Converter

## Business Objective

Allow vendors to convert source videos (MP4, MOV, AVI, etc.) to WebM/VP9+Opus directly in the browser with no server-side processing, using a self-hosted `ffmpeg.wasm` instance. Output is downloaded by the vendor and re-uploaded to their media library.

## Actors

- **Vendor** — selects a quality preset, drops a source video, triggers conversion, downloads the WebM result

## UX Journey

1. Vendor opens `/converter` (standalone page, no theme header/footer)
2. Page loads: FFmpeg wasm assets are fetched and initialized (~31MB, one-time)
3. **Convert view** (default): vendor drops/selects a video file, picks a quality preset (HQ Archive / Balanced / Compressed), clicks "Convert to WebM"
4. Progress bar shows encode % from ffmpeg stderr parse
5. On success: **Result view** slides in with download links (WebM + optional thumbnail), savings summary, "Convert another" button
6. On failure: error message shown inline

## Inputs

| Field | Type | Source |
|---|---|---|
| Source video | File | Browser file picker / drag-drop |
| Quality preset | enum: hq / balanced / compressed | UI radio selector |

Quality → CRF mapping:
- `hq` → CRF 18 (HQ Archive)
- `balanced` → CRF 28
- `compressed` → CRF 40

## Outputs

- `<filename>.webm` — VP9 video + Opus audio, downloaded via Blob URL
- `poster.jpg` — first-frame screenshot, downloaded separately (best-effort)

## Technical Architecture

### Assets (self-hosted, same-origin)
All files live in `ecomcine/modules/tm-video-tools/assets/ffmpeg/`:

| File | Size | Purpose |
|---|---|---|

| `worker.js` | 5 KB | Web Worker entry point |
| `const.js` | 1 KB | Defaults (CORE_URL overridden at runtime) |
| `errors.js` | — | Error types |
| `utils.js` | — | Utilities |
| `types.js` | — | Type stubs |
| `empty.mjs` | — | Module shim |
| `ffmpeg-core.js` | 110 KB | Emscripten JS glue |
| `ffmpeg-core.wasm` | 31 MB | WebAssembly binary |

Package: `@ffmpeg/ffmpeg@0.12.15`, `@ffmpeg/core@0.12.9` (single-threaded build)

### PHP Integration (`tm-video-tools.php`)
- `wp_enqueue_script` — `converter.js` with `type="module"` (via `script_loader_tag` filter), deps: `['jquery']`
- `wp_localize_script` passes: `ffmpegUrl`, `ffmpegCoreUrl`, `ffmpegWasmUrl`, `maxFileSizeMB`, `isLoggedIn`, `loginUrl`

### Encoding pipeline (`converter.js`)
```
initFfmpeg():
  await import(ffmpegUrl)                // dynamic ESM import, same-origin
  new FFmpeg()
  ff.load({ coreURL: ffmpegCoreUrl, wasmURL: ffmpegWasmUrl })

runEncode(file, quality):
  ff.writeFile(inputName, Uint8Array)
  ff.on("log", logHandler)               // captures stderr for error display
  ret = await ff.exec([
    "-i", inputName,
    "-vf", "scale=1280:720:force_original_aspect_ratio=decrease,scale=trunc(iw/2)*2:trunc(ih/2)*2",
    "-c:v", "libvpx-vp9", "-b:v", "0", "-crf", String(crf),
    "-deadline", "realtime", "-cpu-used", "8",
    "-auto-alt-ref", "0", "-lag-in-frames", "0",
    "-tile-columns", "0", "-tile-rows", "0",
    "-pix_fmt", "yuv420p", "-g", "120", "-keyint_min", "120",
    "-ac", "2", "-ar", "44100", "-c:a", "libopus", "-b:a", "96k",
    "-y", outWebm
  ])
  if ret !== 0 → throw (last 6 log lines shown to user)
  ff.readFile(outWebm) → Blob URL download
  posterExtract()                        // best-effort, failure does not block
```

## Phase 2 Work — Attempted & Blocked

### Attempt 1 — Single-threaded wasm memory flags

**Tried**: applied all memory-reduction VP9 flags to `@ffmpeg/core@0.12.9` (ST build):
- `deadline realtime`, `cpu-used 8`, `auto-alt-ref 0`, `lag-in-frames 0`, `tile-columns 0`, `tile-rows 0`, scale cap to 720p

**Result**: still `RuntimeError: memory access out of bounds` at ~8% for typical consumer videos.

**Root cause**: ST wasm heap is hard-capped at 256MB. No ffmpeg flags can work around a wasm heap limit.

---

### Attempt 2 — Multi-threaded wasm (`@ffmpeg/core-mt@0.12.9`) inside WordPress plugin

**Tried**:
- Replaced `ffmpeg-core.js` / `ffmpeg-core.wasm` with MT build from `@ffmpeg/core-mt@0.12.9`
- Added `ffmpeg-core.worker.js` (Emscripten pthread worker)
- Added COOP/COEP headers via WP `send_headers` hook
- Added COOP/COEP headers directly in `templates/page-converter.php` via `header()` calls
- Added `crossOriginIsolated` guard in `converter.js` + user-facing error message
- Passed `workerURL` to `ff.load()`

**Result**: Stuck at "Initialising WebAssembly engine… 90%" — `ff.load()` never resolves or rejects. No error surfaced.

**Root cause**: Not yet confirmed. Likely: the Worker spawned by Emscripten pthreads receives the `workerURL` as a relative path (`./ffmpeg-core.worker.js`), but since `worker.js` is itself served from `/ffmpeg/worker.js`, the relative resolution doubles to `/ffmpeg/ffmpeg/ffmpeg-core.worker.js` (404). Playwright logs confirmed this: `GET /ffmpeg/ffmpeg/ffmpeg-core.js — ERR_ABORTED`.

---

### Attempt 3 — Standalone tool (`tools/webm-converter/`) outside WordPress

**Tried**:
- Created `tools/webm-converter/` as a fully self-contained folder (no WP dependency)
- `server.js` — Node.js server, no npm deps, always sends COOP/COEP + CORP headers
- All ffmpeg MT assets self-contained in `tools/webm-converter/ffmpeg/`
- No file size limits, no login checks, no WordPress at all
- Playwright automated test (`tools/playwright/tests/webm-converter.spec.ts`) confirmed `crossOriginIsolated = true` and file upload working
- Fixed double-path bug: changed `./ffmpeg/ffmpeg-core.js` → `/ffmpeg/ffmpeg-core.js` (absolute URL so worker resolves correctly)

**Result**: Still blocked at `ff.load()` hang / error "Could not load FFmpeg: undefined". The error object has no `.message`, suggesting the inner Worker is crashing silently before it posts back.

**Root cause**: Not yet fully diagnosed. The `ffmpeg-core.worker.js` (Emscripten pthread worker) may be failing to load `ffmpeg-core.js` from inside Service Worker / module worker context, or `SharedArrayBuffer` + Atomics are not fully functional in the Playwright Chromium headless build.

---

## Next Steps (Phase 3 investigation)

**Most promising path**: debug the inner Emscripten worker failure directly.

1. **Test in a real Chrome browser** (non-headless) at `http://localhost:9000` — the Playwright Chromium may have incomplete SharedArrayBuffer support in headless mode. If it works in real Chrome, the issue is Playwright-only.

2. **If still failing in real Chrome**: add `onerror` to `ffmpeg-core.worker.js` to surface the actual inner exception. Edit the file to append:
   ```js
   self.onerror = (e) => { console.error('ffmpeg-core.worker error:', e.message, e.filename, e.lineno); };
   ```

3. **Alternative: try `@ffmpeg/ffmpeg@0.12.x` with `classWorkerURL`** — instead of passing `workerURL` (which goes to Emscripten pthreads), pass `classWorkerURL` to `ff.load()` to override the outer `worker.js` wrapper. This is a different parameter.

4. **Alternative: use `ffmpeg.wasm` v0.11.x (Jerome Wu's original build)** — older API (`createFFmpeg` / `ffmpeg.run()`), different wasm build, may have better-tested MT path.

5. **Alternative: server-side conversion** — upload to WP server via REST API, run `ffmpeg` binary in a Docker container, return WebM. Removes all wasm complexity; adds upload UX.
